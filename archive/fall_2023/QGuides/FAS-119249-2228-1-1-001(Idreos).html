One of the best courses, if not the best course, I've taken at Harvard. If you're thinking about taking the course, do it! Find some friends to take it with you, clear space in your schedule, and deep dive into the course. It's definitely worth it; don't be deterred by any reviews. A few reasons the class stands out:

1) Stratos and his lab are some of the most chill, yet cracked people I've ever met—just look at the awards they win. This translates to easy–to–understand lectures and a good introduction to research. Lectures teach you how to walk step–by–step through the different operators you learn about, as well as how to break down any high–level database concept into the fundamentals: cache utilization, branching, etc. (I would suggest taking CS 61 before this course; I will note that I found the project easier than the psets in CS 61). Furthermore, doing database research is comparable to the work you do in lectures.

2) You learn how to code—independently. Unlike in CS 61, where the projects were contained to 1–2 weeks of work, no one besides you in this class (not the TFs nor Stratos) will know your entire database implementation because you'll be working on it throughout the semester. Besides the handout code—which you can revamp to your heart's content—you decide which functions to create and how they work, how to parse inputs, and how to return results. Attending lectures should provide a rough outline for you to follow, but besides that, you learn how to debug your code. (Please, please, please install Valgrind immediately after starting the course. I promise you'll save 2 weeks' worth of time you'd otherwise spend debugging weird errors and dealing with memory management in C. You can either launch an EC2 instance or run it through Docker). Modularize your functions, unit test new operators, and spend time thinking about the high–level concepts (you should really only spend 30% of your time on this project actually coding); this project isn't bad once you realize how to break a big task into a collection of smaller ones.

3) The grading is very generous with extra credit, namely, on the midterms. You don't need to sweat your grade!
Do not procrastinate on the project. It is physically impossible to complete in less than a month.
This class made me far better of a software engineer and database system designer. Stratos is an incredible professor – terrific lectures and understanding of the subject. But DO NOT TAKE THIS CLASS unless you want to forgo all sleep in November and December. The workload of this class is absolutely no joke – the project, even if you paced yourself over the course of the semester, is impossible to finish; there's also the time you need to review and master concepts from lecture first, study for + do the take–home midterms (most students spend 5 –10 hours on each midterm), and additionally do the readings + watch sections to improve your understanding (most students gave up on this).

It doesn't matter how prepared for this class you are; some students had done 61/161 already, some had done only 61, I had not done any systems class before but excelled at 124; all of us struggled to finish the project.
Expect to:
– Write 4,000 lines of C code.
– Know the basics of column–style data bases.
– Understand how data is laid out on disk and what major considerations are of performant database systems (e.g., branch mispredictions, the many types of indices, key–value store's vs. column–stores vs. row–stores).
– Spend 20 hours on both the take home midterms only to get your 120% out of 150% with bonus rounded down to a 100% so that everyone has the same grade.
– Be very confused about what exactly is required in the final report, whether it is 8 total experiments or 4 experiments for each milestone plus three work load experiments (it is 2 experiments per milestone and any workload experiments being bonuses, by the way).
– Be very confused about how to implement many of the sentences on the Project page surrounding the milestone, with little guidance, besides StackOverflow and Github Copilot.
– Somewhat compete with other people in the class, in the sense that they will likely be very guarded with you, since performance in the class is a function of "how much you can get done by the deadline"
– Be confused about how the grading server tests (tests 1–3 use 10K rows, 4–9 use 100K, 10–37 use 1M, and you get credit for a test so long as you pass it at least once).
– Spend many days alone debugging C.
– Feel like a database boss at the end of it :)
– Be able to read industry papers (e.g., the Snowflake, Databricks, or Kafka papers) and almost fully understand what is going on and what the major considerations are, in addition to the tradeoffs that the authors are making and potential for the systems to be improved.
– Learn a lot of ways to make code more performant at scale, without compiler optimizations necessarily, like avoiding IF statements for branching, linked lists for random accesses, large data structures for cache misses, etc.
– Learn how to profile code on your hardware and tune your system to best perform given your system.
– Debug in containers.
Very open ended class and the value you get is proportional to how much work you put in. You can get so much valuable knowledge and experience out of the class if you put full effort into the project and reading all the research papers.
